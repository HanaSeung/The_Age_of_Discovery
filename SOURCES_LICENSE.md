# 지도 데이터 출처 및 라이선스 (Map Data Source & License)

## 사용 데이터
- **이름**: Natural Earth — 1:50m Physical Vectors, "land" (해안선/육지 폴리곤)
- **원본 제공처**: https://www.naturalearthdata.com/
- **취득 경로(미러)**: nvkelso/natural-earth-vector (GitHub)
  `geojson/ne_50m_land.geojson`
- **라이선스**: **퍼블릭 도메인 (Public Domain)**
  > "Natural Earth is a public domain map dataset... No permission is needed
  > to use Natural Earth. Crediting the authors is unnecessary."
  (출처를 기록해 두는 것은 권장 사항이며 의무는 아님)

## 가공 내역 (재현 가능)
1. `ne_50m_land.geojson` (원본 60,669점) 다운로드
2. Douglas–Peucker 단순화: **tolerance = 0.08°** (shapely `simplify`, `preserve_topology=True`)
3. 면적 < 0.01 deg² 인 미세 폴리곤 제거 (노이즈 섬 정리)
4. 좌표 소수점 2자리 반올림(약 1km 정밀도) → 최종 **931 폴리곤 / ~15,000점 / 220KB**
5. 결과를 게임 HTML에 인라인 임베드 (외부 네트워크 의존 없음)

## 사용하지 않은 것
- 국가 경계(admin) · 현대 도시명 · 도로 등은 **포함하지 않음** (요구사항 13)
- 해안선/대륙 형태는 임의로 그리지 않고 위 실측 데이터만 단순화하여 사용 (요구사항 11)

# 해류 데이터 출처 및 라이선스 (Ocean Current Data)

## 사용 데이터
- **이름**: NASA **OSCAR** (Ocean Surface Current Analysis Real-time) 표면 해류
- **일자**: 2014-01-31 (단일 스냅샷, 연중 대순환 패턴 대표)
- **원 제공처**: NASA / JPL PODAAC — https://podaac.jpl.nasa.gov/
- **라이선스**: **퍼블릭 도메인** (NASA 자료는 저작권 없음)
- **취득 경로(JSON 재배포)**: `cambecc/earth` (GitHub, **MIT License**)
  `public/data/oscar/20140131-surface-currents-oscar-0.33.json`
- **원 격자**: 1080 × 481 (0.333°), u/v 성분, 단위 m/s, 육지 = null

## 가공 내역 (재현 가능 — `build_currents.py`)
1. 원본 JSON 다운로드 (5.2 MB, `oscar_raw.json` 로 캐시)
2. **2° 격자(180 × 90)로 박스 평균** 다운샘플 — null(육지) 셀은 평균에서 제외
3. 데이터 없는 셀(육지/결측)은 u = v = 0
4. **양자화**: m/s → cm/s → 2 cm/s 단위 int8 (범위 ±2.54 m/s)
5. u 전체(16,200) + v 전체(16,200) = 32,400 byte → base64 임베드 (`currents_data.js`, 42 KB)
6. **정규화 기준** `refMs` = 해양셀 유속의 **p99 = 0.746 m/s** 를 함께 기록

## 좌표계 · 세기 스케일 (게임 적용)
- 지도와 동일: equirectangular, 경도 [-180,180] → x, 위도 [90,-90] → y
- 셀 인덱스: `ti = (lon+180)/2`, `tj = (90-lat)/2`. **쌍선형 보간**, 경도는 wrap, 위도는 clamp
- 화면 y축은 남쪽이 +이므로 북향 성분 v는 **부호 반전**하여 적용
- **세기 환산**: `월드px/초 = (u_m/s ÷ refMs) × CUR_MAX`, `CUR_MAX = 94`
  = 선박 최고속 `SPEED(235)`의 **40%**. 즉 p99 해류가 선박 최고속의 40%
- 육지에서는 게임 육지마스크(`isLand`)로 한 번 더 0 처리

## 적용식 (선박)
선박 속도는 **물에 대한 상대속도**, 해류는 물 자체의 이동으로 처리:
```
위치 += (v_ship + v_current) · dt
```
기존 마찰·속도상한은 `v_ship`에만 적용되므로 자연스럽게 결합된다
(순류 → 실효 속도 증가, 역류 → 감소, 횡류 → 표류).

## 검증 (`verify_currents.py`, 8/8 통과)
생성된 `currents_data.js` 를 역디코딩해 실제 해류와 대조:

# 축척 개편 (2차) — 해안선 고정밀화 + 배율 확장

## 배경
벡터 지도라도 **단순화한 데이터가 확대의 천장**을 만든다. 기존 0.08° 단순화는
오차 8.9 km(1.82 월드px)로, 오차를 화면 3px 이하로 유지하려면 배율 1.6이 한계였다
(당시 ZMAX 2.2는 이미 한계를 넘긴 상태). 배 아이콘도 화면 고정 33px이라
최대 줌인에서도 실제 74 km — 축척상 이탈리아 반도 크기였다.

## 변경 내용
| 항목 | 이전 | 현재 |
|---|---|---|
| 해안선 데이터 | 0.08° 단순화 · 931폴리곤 · 15,000점 | **단순화 없음** · **1,406링** · **60,527점** |
| 좌표 정밀도 | 0.01° 반올림 (8.9 km 실효오차) | **0.01° = 1.11 km** (실효) |
| 저장 형식 | HTML 인라인 JSON 220 KB | **int16 이진 + base64** → `land_data.js` 322 KB |
| 충돌 마스크 | 4096×2048 (셀 9.8 km) | **8192×4096** (셀 **4.89 km**) |
| 최대 배율 ZMAX | 2.2 | **13** |
| 배 아이콘 | 33 화면px | 33 화면px (가독성 위해 유지) |
| HTML 크기 | 239 KB | **19 KB** (데이터 분리) |

## 지도 축척 (확정)
- 월드 8192×4096 px = 지구 전체 → **적도에서 1 월드px = 4.89 km**
- equirectangular 특성상 가로 축척은 위도에 따라 축소 (60°N에서 2.45 km/px)

## 배 아이콘 실제 크기 (아이콘 33 화면px 고정)
| 배율 | 아이콘 환산 크기 |
|---|---|
| 최대 줌아웃 | 834 km |
| 초기 0.42 | 388 km |
| 배율 5 | 32.6 km |
| **최대 ZMAX 13** | **12.5 km** (이전 ZMAX 2.2에서 74 km — **6배 개선**) |

실제 16C 갈레온 50m는 이 축척에서 0.01px로 표현 불가.
따라서 **선박은 축척 물체가 아니라 해도 기호(chart symbol)** 로 규정한다.
아이콘을 12px로 줄이면 최대 배율에서 4.5 km까지 내려가지만 화면상 너무 작아
식별이 어려워, **가독성을 우선해 33px를 유지**한다. 축척 개선은 ZMAX 확장으로만 얻는다.

## 데이터 생성 (재현 가능 — `build_land.py`)
1. `ne_50m_land.geojson` 다운로드 (1.6 MB, 캐시)
2. **단순화 없이** 전 좌표 사용, 0.01° 반올림 후 연속 중복점 제거
3. 외곽링 면적 < 0.001 deg² 폴리곤 제외 (노이즈 섬)
4. 인코딩: `[uint32 링수][uint32 링길이×N][int16 lon*100, int16 lat*100 × P]`
   → 247,736 byte → base64 → `land_data.js`
5. 브라우저에서 DataView로 디코드해 단일 Path2D 구성 (evenodd)

## 충돌 마스크
8192×4096 캔버스를 한 번에 `getImageData` 하면 순간 134 MB가 필요하므로,
**512행 밴드 단위**로 래스터화·판독한다 (피크 16.8 MB).
최종 `Uint8Array` 32 MB 상주.

## 검증 (`verify_scale.js`)
- 링 수 1,406 / 점 60,527 — 헤더 선언값과 **합계 일치** ✔
- 바이트 정합 247,736 = 헤더 + 점×4 ✔
- 좌표 범위 경도 −180~180 / 위도 −90~83.6 ✔
- ZMAX 13에서 해안선 오차 **2.9 화면px** — 목표(3px) 이내 ✔

## 남은 한계
더 확대하려면 10m급 원본(오차 0.22 km, 배율 66까지)이 필요하나,
용량이 수 MB로 늘어 배율별 데이터 교체(LOD)가 필수가 된다. 현재는 미적용.

# 수심 데이터 출처 및 라이선스 (Bathymetry)

## 사용 데이터
- **이름**: Natural Earth **1:10m Bathymetry** (등심선 폴리곤)
- **원 제공처**: https://www.naturalearthdata.com/
- **라이선스**: **퍼블릭 도메인 (Public Domain)**
- **취득 경로**: nvkelso/natural-earth-vector (GitHub), `geojson/ne_10m_bathymetry_*.geojson`
- 사용 파일: `K_200`, `J_1000`, `G_4000`, `E_6000`

## 중요 — 데이터가 누적(nested) 구조다
각 파일은 "해당 수심보다 **깊은** 전체 영역"이다. 따라서 면적이 단조 감소한다
(200m 38,876 → 1000m 35,875 → 4000m 19,062 → 6000m 231 deg²).
**요구사항 4는 "깊은 것부터 얕은 것 순"으로 칠하라고 했으나, 이 구조에서는 불가능하다.**
얕은 밴드가 깊은 밴드를 완전히 포함하므로, 나중에 칠하면 깊은 곳을 덮어버린다.
실제 구현은 **얕은 것부터 깊은 것 순**으로 덮어 칠한다:

```
① 월드 전체를 0–200m 색으로 채움
② 200m 폴리곤을 200–1000m 색으로 덮음   → 200m 미만만 ①색으로 남음
③ 1000m 폴리곤을 1000–4000m 색으로 덮음
④ 4000m → 심해평원색,  ⑤ 6000m → 해구색
⑥ 등심선 윤곽(세피아 0.20 알파) → ⑦ 나침선·경위선 → ⑧ 육지 → ⑨ 해안선
```
육지를 **맨 나중에** 칠하므로, 등심선이 해안을 넘어 육지로 삐져나와도 육지가 덮어 가린다.
요구사항 6의 "육지-바다 경계 어긋남"은 이 순서로 구조적으로 해결된다.

## 선택한 등심선 단계 (5구역)
| 구역 | 범위 | 색 | 선택 이유 |
|---|---|---|---|
| 연안·대륙붕 | 0–200 m | `#bdd2c4` | 200m는 **대륙붕단**의 국제 표준 경계. 북해·순다붕·황해 등 항해상 의미가 큰 얕은 바다를 한 덩어리로 잡아준다 |
| 대륙사면 | 200–1000 m | `#a4c1b5` | 대륙붕에서 심해로 떨어지는 급사면 구간 |
| 심해 | 1000–4000 m | `#8bb0a8` | 대양의 기본면. **기존 바다색을 그대로 둬** 전체 인상이 바뀌지 않게 함 |
| 심해평원 | 4000–6000 m | `#6f948e` | 태평양·대서양 심해분지의 실제 주 수심대 |
| 해구 | 6000 m+ | `#557a76` | 일본·마리아나·통가 해구. 이게 없으면 "가장 깊은 곳"이 표현되지 않는다 |

2000m, 3000m 등은 의도적으로 뺐다. 11단계를 다 쓰면 색 차이가 미세해져
고지도풍 **뚜렷한 밴드** 인상이 무너지고 현대 지도앱처럼 보인다.

## 팔레트 설계
기존 팔레트(청록 바다 `#8bb0a8` · 오커 육지 `#d8bf8e` · 세피아 해안선 `#6b4a2a`)를 유지하되,
**기존 바다색을 심해(1000–4000m) 자리에 고정**하고 위아래로 확장했다.
얕은 쪽은 밝고 따뜻하게 가서 모래색 육지와 자연히 이어지고, 깊은 쪽은 채도를 낮춰 가라앉는다.
등심선 윤곽은 세피아 20% — 동판화 해도의 새김선 느낌만 주고 도드라지지 않게.

**제거함**: 기존 바다에 깔려 있던 **위도 기반 세로 그라데이션**. 수심과 무관한 장식이었고,
극지방을 어둡게 만들어 수심 색과 충돌했다.

## 가공 내역 (재현 가능 — `build_depth.py`, 외부 패키지 불필요)
1. 4개 GeoJSON 다운로드 (약 20 MB, 캐시)
2. **Douglas–Peucker 단순화 tolerance 0.03° (= 3.34 km)** — 반복형 구현(재귀 깊이 회피)
3. 폴리곤 면적 필터: 200m·1000m 0.5 deg², 4000m 0.8 deg², **6000m 0.05 deg²**
   (해구는 폭이 좁아 일반 기준으로는 통째로 사라짐 — 실제로 초기 시험에서 일본해구가 소실됐다)
4. 좌표 0.01° 반올림 → int16, 연속 중복점 제거
5. 인코딩: `[uint32 밴드수][밴드별 depth+링수][전 링 길이][int16 lon*100, lat*100]`
   → **985 링 / 104,224 점 / 411 KB 이진 → base64 548 KB** (`bathy_data.js`)
6. 밴드 내 폴리곤 **중복 0.00%** 확인 → 합집합 연산 불필요, evenodd 채움으로 구멍까지 정확히 처리

## 배율 한계
- 단순화 오차 3.34 km = **0.68 월드px**
- 밴드 경계가 화면 3px 이내로 유지되는 **한계 배율 ≈ 4.4** (해안선 `land_data`는 13.2)
- ZMAX 13까지 확대하면 밴드 경계는 약 9 화면px 오차로 다소 완만해진다. 다만
  ① 등심선은 해안선과 달리 **부드러운 경계**라 눈이 오차를 검증할 기준이 없고,
  ② 고배율에서는 대개 한 밴드 **안쪽**만 화면에 들어와 경계 자체가 보이지 않으며,
  ③ 해안 부근 어긋남은 육지가 덮어 가린다.
  따라서 실사용상 문제가 없다고 판단해 ZMAX는 13으로 유지했다.
  더 정밀히 하려면 tolerance 0.01°로 낮출 수 있으나 용량이 약 3배로 늘어난다.

  (예: 대서양 45°W 30°N은 중앙해령 위라 실제로 얕은 것이 정답이었다)

## 스코프

# 강수·기온 데이터 출처 및 라이선스 (Precipitation & Air Temperature)

## 사용 데이터
- **이름**: NOAA PSL, **NCEP/NCAR Reanalysis 1** Monthly Long-Term Mean (1991–2020)
  - 강수: `prate.sfc.mon.ltm.nc` (지표 강수율, kg/m²/s)
  - 기온: `air.2m.mon.ltm.nc` (지상 2m 기온, K)
- **원 제공처**: NOAA Physical Sciences Laboratory — https://psl.noaa.gov/
- **라이선스**: **퍼블릭 도메인** (미국 정부 저작물)
- **취득 경로**: OPeNDAP ASCII (`.ascii?` 질의) — 외부 라이브러리 불필요
- **원 격자**: T62 가우시안 192 × 94, 12개월

바람(`build_wind.py`)·구름(`build_cloud.py`)과 같은 창구·같은 축소 방식이라
네 데이터의 4° 격자 칸이 정확히 겹친다.

## 왜 운량이 아니라 강수량을 따로 받았는가

처음에는 강수를 **운량에서 유도**했다. 구름이 두꺼운 곳에 비가 온다는 전제였다.
검증에서 결과가 **정반대로** 나왔다.

| 바다 (7월) | 운량 | 실제 강수 |
|---|---:|---:|
| 페루 앞바다 (20°S 90°W) | 66.5 % | **0.20 mm/일** |
| 적도수렴대 대서양 (8°N 25°W) | 60.1 % | **5.20 mm/일** |

아열대 동안(페루·나미비아·캘리포니아·카나리아)에는 해양성 **층적운**이 두껍게
깔려 하늘을 덮지만 비는 거의 내리지 않는다. 지구에서 가장 메마른 바다가
운량으로는 다우대보다 흐리게 나온다. 하늘이 덮인 것과 비가 오는 것은 다르다.

## 왜 수온 근사가 아니라 실측 기온을 받았는가

비가 눈이 되는지를 정하는 것은 **물 온도가 아니라 공기 온도**다. 처음에는
위도·계절로 지어낸 수온 근사(`seaTempC`)를 대용으로 썼는데, 대용품의 오차와
근사의 오차가 겹쳐 고위도에서 3~6℃씩, 그것도 **부호가 양쪽으로** 어긋났다.

| 지점 | 근사 | 실측 | 오차 |
|---|---:|---:|---:|
| 60°N 1월 | 0.4 | 6.5 | −6.1 |
| 72°N 7월 | 1.1 | 4.6 | −3.5 |
| 55°S 1월 | 7.4 | 0.8 | +6.6 |
| 65°S 7월 | 0.4 | −23.2 | +23.6 |

이 위에 눈 문턱을 얹으면 시험한 지점만 맞고 나머지는 틀린다. 실측으로 바꾸니
**위도만으로는 나올 수 없는 것**까지 따라왔다 — 1월 같은 위도에서
노르웨이해 **+5.8℃**(만류), 래브라도해 **−3.5℃**(한류). 9.3℃ 차이다.

## 가공 내역 (재현 가능 — `build_precip.py` / `build_airtemp.py`)
1. `.dds` 로 격자 크기를 먼저 확인 (창구가 바뀌면 여기서 걸린다)
2. `.ascii` 전체 다운로드 후 원문 캐시 (`precip_raw.txt` 2.9 MB / `airtemp_raw.txt` 2.3 MB)
3. 단위 환산 — 강수 `× 86400` → mm/일, 기온 `− 273.15` → ℃ (켈빈 여부는 평균값으로 자동 판별)
4. 가우시안 192×94 → 등간격 **4° 90×45** 박스 평균 (결측 셀 제외)
5. 양자화 → base64 임베드, 각 **48,600 byte / 64 KB**
   - 강수: `0.1 mm/일` 단위 uint8 (0 ~ 25.5). 최대 18.92 이므로 잘림 없음
   - 기온: `baseC −75℃` 에서 `0.5℃` 단위 uint8 (−75 ~ +52.5).
     최저 −70.8 / 최고 37.0 이므로 잘림 없음 (초기 −60 기준에서는 남극 내륙 361칸이 잘렸다)
6. 정규화 기준 기록 — 강수 `p50 1.80 / p90 5.46 / p99 9.86 mm/일`, 기온 `minC / maxC`

## 게임 적용 (`precipAt`)
두 겹을 곱한다.
```
세기 = 그 바다가 원래 비가 잦은가 (실측 mm/일)  ×  지금 구름 밑인가 (화면의 구름)
종류 = 실측 지상기온 < 1.5℃ 이면 눈, 아니면 비
```
- `PRECIP_MIN 0.5 mm/일` 아래는 비로 치지 않음, `PRECIP_FULL 10 mm/일`(≈p99)에서 세기 1
  - p90 으로 자르면 적도수렴대와 몬순이 통째로 상한에 붙어 구별이 사라진다
- `PRECIP_ST 1.5℃` — 0℃ 가 아닌 이유는 눈이 떨어지며 녹기 때문. 기온이 실측이라
  이 값을 근사 오차를 메우는 손잡이가 아니라 물리적 근거대로 잡을 수 있다
- 구름 표시를 끄면 비도 그친다 (`cloudOpacityAt` 을 거치므로). 의도한 동작이다

`seaTempC`(위도 수온 근사)는 지우지 않고 남겨 두었다. 기온 자료가 없을 때의
대비책이자, 나중에 해빙이나 수온 자체가 필요할 때 쓸 자리다.
